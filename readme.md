# CUDA Einstein Candidate Second-Pass Validator

This repository contains experimental tools for investigating connected 24-triangle tile candidates on a triangular lattice.

The original CUDA validator performs second-pass finite validation on candidate monotiles generated by a CUDA aperiodic monotile hunter. Additional tools provide per-case periodic torus audits, periodic-witness export and verification, independent checks, forced-structure probes, and motif-stability experiments.

> Important: these tools are not mathematical proof engines. A strong finite result does not prove that a tile is an einstein tile. It only provides finite computational evidence that a candidate may be worth further investigation.
>
> Conversely, one independently verified periodic torus tiling is decisive: it proves that the candidate is not aperiodic under the transformations allowed by that test.

## Purpose

The project is used to test whether generated candidate tiles:

* can be rejected because a periodic tiling is found;
* can grow large finite patches without obvious periodic behaviour;
* show strong boundary or forced-placement patch growth;
* survive finite torus searches up to a chosen period;
* show potentially interesting local forcing or repeated motifs;
* deserve deeper analysis as possible aperiodic monotile candidates.

The CUDA validator is especially useful for filtering a large number of generated candidates down to a smaller set of candidates worth stronger investigation.

## Input Format

The validator expects candidate files named like:

```text
candidate_0000858.txt
candidate_0000148.txt
candidate_0001633.txt
```

It does **not** validate SVG files directly.

Correct input:

```text
candidate_0000858.txt
```

Incorrect input:

```text
candidate_0000858_tile.svg
candidate_0000858_best_patch.svg
candidate_0000858_validation.txt
```

A candidate file should contain normalised tile cells, for example:

```text
normalized_cells x y orientation:
0 4 0
0 4 1
1 1 0
...
```

Each row represents a unit triangle:

```text
x y orientation
```

where `orientation` is either `0` or `1`.

To validate one candidate, place only that candidate file into its own input folder:

```bash
mkdir -p single_candidate
cp candidate_0000858.txt single_candidate/
```

## Compilation

### CUDA Validator

Compile with `nvcc`:

```bash
nvcc -O3 -std=c++17 -arch=sm_80 --use_fast_math \
  einstein_validator_v1.cu \
  -o einstein_validator_v1
```

Change `sm_80` if the target GPU has a different CUDA architecture.

If a matching Makefile is available:

```bash
make
```

### Torus Audit

Compile the torus audit with a standard C++ compiler:

```bash
g++ -O3 -std=c++17 -Wall -Wextra -pedantic \
  einstein_torus_audit.cpp \
  -o einstein_torus_audit
```

## What the CUDA Validator Tests

The validator performs several finite checks on each candidate.

### 1. Periodic Torus Search

The validator searches finite repeating torus regions up to the selected period.

If a candidate can tile one of these finite wrapped regions exactly, then it has a periodic tiling and is rejected.

Relevant output fields:

```text
periodic_found
periodic_W
periodic_H
periodic_limit_hit
torus_nodes
```

Interpretation:

```text
periodic_found: 1
```

means a periodic tiling was found, so the candidate is not an einstein candidate under the transformations allowed by that run.

```text
periodic_found: 0
periodic_limit_hit: 0
```

means no periodic tiling was found in the tested finite range, and the search completed within the node limit.

```text
periodic_limit_hit: 1
```

means at least one search reached the node limit, so the result is incomplete for that tested range.

### 2. No-Reflection Periodic Search

The validator can also test candidates without allowing reflected copies.

Relevant output fields:

```text
no_reflection_periodic_found
no_reflection_periodic_W
no_reflection_periodic_H
```

This is useful because reflection-allowed and no-reflection-only tiling are different research targets.

### 3. Patch Growth Tests

The validator grows finite patches using three methods:

```text
random_patch_tiles
boundary_patch_tiles
forced_patch_tiles
```

The strongest result is recorded as:

```text
best_patch_method
best_patch_tiles
```

The methods are:

* `random`: random greedy patch growth;
* `boundary`: prioritises filling open boundary cells;
* `forced`: searches for cells where only one valid placement is currently possible, then falls back to boundary placement.

A high `forced_patch_tiles` value can be interesting because it suggests that local geometry may be strongly constraining patch construction.

However, a large finite patch does not prove aperiodicity.

### 4. Exact Hex Region Test

The validator can also try to exactly fill finite hex-shaped regions.

Relevant output fields:

```text
exact_region_found
exact_region_radius
exact_region_cells
exact_region_limit_hit
exact_region_nodes
```

If exact regions are found, that provides further finite evidence that the tile can cover structured finite areas. It is still not proof of infinite tiling.

## Classification Labels

The CUDA validator sorts results into classification folders.

### `periodic_reject`

A periodic tiling was found. The candidate should be rejected as an einstein candidate under the transformations allowed by that test.

### `weak_patch_reject`

No periodic tiling was found in the finite test, but patch growth was weak.

### `unstable_candidate`

The candidate survived some tests but did not reach the stronger patch threshold.

### `strong_patch_candidate`

The candidate survived the finite periodic test and grew a large patch. These candidates are worth saving and reviewing.

### `very_strong_candidate`

The candidate grew a very large patch and also passed additional exact-region criteria. These are the strongest finite-search candidates, but they are still not proven einstein tiles.

## Per-Case Torus Audit

`einstein_torus_audit` tests rectangular wrapped torus cases individually and writes one CSV row after each case.

This makes it possible to see exactly which dimensions:

* completed with no periodic tiling;
* found a periodic tiling;
* reached the node limit.

Example period-22 audit:

```bash
./einstein_torus_audit \
  --input candidate_0000858.txt \
  --out audit_period22 \
  --max-period 22 \
  --nodes 50000000 \
  --mode both \
  --progress-sec 15
```

Example stronger period-60 screen:

```bash
./einstein_torus_audit \
  --input candidate_0000858.txt \
  --out audit_period60 \
  --max-period 60 \
  --nodes 100000000 \
  --mode both \
  --progress-sec 30
```

The audit CSV includes fields such as:

```text
time
mode
W
H
board_cells
tile_cells
orientations
placements
nodes
node_limit
status
elapsed_seconds
```

Possible statuses:

```text
NO_TILING
FOUND
LIMIT
```

`NO_TILING` means that exact rectangular torus case completed without finding a periodic tiling.

`LIMIT` means the node cap was reached before the case was resolved.

`FOUND` means the solver constructed a periodic torus tiling. A periodic witness should then be exported and independently verified.

## Positive Controls

Before relying on negative torus-audit results, the audit should be tested with known periodic tiles.

### Single-Triangle Control

A one-triangle tile should tile every compatible torus periodically:

```bash
./einstein_torus_audit \
  --input test_single_triangle.txt \
  --out audit_single_triangle \
  --max-period 4 \
  --nodes 100000 \
  --mode reflect
```

Expected result:

```text
FOUND
```

### Periodic 24-Cell Block Control

A deliberately periodic 24-cell block checks that the audit can detect periodicity for a tile with the same area as the main candidates:

```bash
./einstein_torus_audit \
  --input test_periodic_24block.txt \
  --out audit_periodic_24block \
  --case 6 8 \
  --nodes 1000000 \
  --mode both
```

Expected result:

```text
reflect: FOUND
no_reflect: FOUND
```

These controls demonstrate that the torus audit can produce `FOUND` for genuinely periodic examples.

## Exporting and Verifying a Periodic Witness

A `FOUND` result should not be treated as final until the specific witness has been exported and checked independently.

Export a witness for one case:

```bash
./einstein_torus_audit \
  --input candidate_0000858.txt \
  --out witness_output \
  --case 8 48 \
  --nodes 100000000 \
  --mode reflect \
  --save-witness \
  --progress-sec 5
```

This exports files including:

```text
periodic_witness_reflect_W8_H48_placements.csv
periodic_witness_reflect_W8_H48_cells.csv
```

Verify the witness independently:

```bash
python3 verify_torus_witness.py \
  --input candidate_0000858.txt \
  --placements witness_output/periodic_witness_reflect_W8_H48_placements.csv \
  --cells witness_output/periodic_witness_reflect_W8_H48_cells.csv \
  --W 8 \
  --H 48 \
  --mode reflect \
  --report witness_output/independent_verification.txt
```

A decisive verified result looks like:

```text
placements_checked: 32
expected_placements: 32
cells_covered_once: 768
missing_cells: 0
multiply_covered_cells: 0
RESULT: PASS
```

A verified periodic torus witness can be repeated indefinitely in both directions, producing a periodic plane tiling.

## Forced Structure Probe

`forced_structure_probe.py` constructs finite patches and records local forcing statistics, boundary option counts, repeated clusters, and SVG snapshots.

Example:

```bash
python3 forced_structure_probe.py candidate_0000858.txt \
  --out structure_probe_output \
  --radius 70 \
  --trials 3 \
  --max-tiles 1200 \
  --seed 858 \
  --allow-reflections 1 \
  --heartbeat-sec 10
```

Important limitation:

A locally unique move inside a finite bounded patch does not prove global forcing. The finite boundary may remove alternatives that would exist in an infinite tiling.

Forcing statistics are exploratory evidence, not proof.

## Motif Stability Probe

`motif_stability_probe.py` checks whether motifs recur across independently built patches with different seeds and growth policies.

Example:

```bash
python3 motif_stability_probe.py candidate_0000858.txt \
  --out motif_stability_output \
  --seeds 858,271828,314159 \
  --modes forced,mixed,random \
  --reflection-modes 1 \
  --radius 60 \
  --max-tiles 800 \
  --motif-radii 3,4,5,6 \
  --stability-fraction 0.67 \
  --heartbeat-sec 10
```

A motif appearing in one patch is not enough to establish a substitution structure. The meaningful result is a motif that persists across independent seeds and construction policies.

## Independent Python Test

`independent_tile_test_progress.py` provides an additional implementation path for finite patch and periodicity checks.

Example:

```bash
python3 independent_tile_test_progress.py \
  candidate_0000858.txt \
  --max-period 16 \
  --nodes 1000000 \
  --radius 90 \
  --trials 250 \
  --seed 858 \
  --heartbeat-sec 30 \
  --progress-every 10
```

Independent implementations are useful because they reduce dependence on one parser, orientation generator, or search routine.

## Interpreting Results

A promising finite result may look like:

```text
classification: strong_patch_candidate
periodic_found: 0
periodic_limit_hit: 0
forced_patch_tiles: high
best_patch_method: forced
best_patch_tiles: high
```

This means the candidate survived the validator's tested finite periodic search and grew a large patch.

It does **not** prove that the tile is an einstein tile.

Use the following rules:

```text
Large finite patch:
    Interesting finite evidence, not proof.

NO_TILING:
    Rules out only the tested finite torus case.

LIMIT:
    Inconclusive for that case.

Verified FOUND:
    Decisively proves periodic tiling under the tested transformation rules.

Reflection-allowed FOUND:
    Disqualifies a candidate when reflected copies are permitted.

No-reflection FOUND:
    Disqualifies a candidate even in the stricter no-reflection setting.
```

## Research Findings

### Candidate `0000858`

Candidate `0000858` initially produced strong finite patch-growth results, including a patch of 10,539 tiles.

The torus audit later found a reflection-allowed periodic tiling on an `8 × 48` wrapped board. Its exported witness was independently verified:

```text
placements_checked: 32
expected_placements: 32
cells_covered_once: 768
missing_cells: 0
multiply_covered_cells: 0
RESULT: PASS
```

Therefore, candidate `0000858` is not an aperiodic monotile when reflected copies are allowed.

The no-reflection-only version remains a separate question.

### Candidate `0003948`

Candidate `0003948` also produced promising finite results. The rectangular torus audit found reflection-allowed periodic hits beginning at `6 × 24`.

These hits should be treated as provisional until a witness is exported and independently verified with `verify_torus_witness.py`.

## What Counts as Proof?

This project provides finite computational evidence and periodic counterexamples. It is not a proof system.

To prove that a candidate is an einstein tile, further work would be needed:

* a mathematical proof that the tile admits at least one infinite plane tiling;
* a proof that every valid plane tiling is non-periodic;
* a substitution or hierarchical structure;
* a globally forced hierarchy argument;
* exclusion of all possible periodic translation lattices, including skew period bases;
* a mathematically checkable proof or certificate.

## Notes

* The validator can run for many hours or days on deep settings.
* `--period` and `--nodes` have a major effect on runtime.
* Larger `--period` values can greatly increase the search space.
* Deep validation should only be used for the strongest candidates.
* The safest workflow is staged testing: fast batch validation, medium validation, torus screening, then deep single-candidate investigation.
