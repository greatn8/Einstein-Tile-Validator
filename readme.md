# CUDA Einstein Candidate Second-Pass Validator

This program performs second-pass finite validation on candidate monotiles generated by a CUDA aperiodic monotile hunter.

It takes candidate `.txt` files, runs stronger finite checks, and classifies each candidate based on periodic tiling tests, patch-growth behaviour, and exact finite-region filling tests.

> Important: this validator is **not a mathematical proof engine**. A strong result does not prove that a tile is an einstein tile. It only provides finite computational evidence that a candidate is worth further investigation.

## Purpose

The validator is used to test whether generated candidate tiles:

* can be rejected because a periodic tiling is found;
* can grow large patches without obvious periodic behaviour;
* show strong boundary or forced-placement patch growth;
* survive finite torus searches up to a chosen period;
* deserve deeper analysis as possible aperiodic monotile candidates.

The program is especially useful for filtering a large number of generated candidates down to a smaller set of strong candidates.

## What the Validator Tests

The validator performs several checks on each candidate.

### 1. Periodic Torus Search

The program searches finite repeating torus regions up to the selected period.

If a candidate can tile one of these finite wrapped regions exactly, then it has a periodic tiling and is rejected.

Relevant output fields:

```text
periodic_found
periodic_W
periodic_H
periodic_limit_hit
torus_nodes
```

Interpretation:

```text
periodic_found: 1
```

means a periodic tiling was found, so the candidate is not an einstein candidate.

```text
periodic_found: 0
periodic_limit_hit: 0
```

means no periodic tiling was found in the tested finite range, and the search completed within the node limit.

```text
periodic_limit_hit: 1
```

means the search hit the node limit, so the result is weaker and inconclusive for that tested range.

### 2. No-Reflection Periodic Search

The validator can also test the candidate without allowing reflected copies.

Relevant output fields:

```text
no_reflection_periodic_found
no_reflection_periodic_W
no_reflection_periodic_H
```

This is useful for checking whether a candidate behaves differently in chiral/no-reflection mode.

### 3. Patch Growth Tests

The validator grows finite patches using three methods:

```text
random_patch_tiles
boundary_patch_tiles
forced_patch_tiles
```

The strongest result is recorded as:

```text
best_patch_method
best_patch_tiles
```

The three patch methods mean:

* `random`: tries random greedy patch growth;
* `boundary`: prioritises filling open boundary cells;
* `forced`: looks for cells where only one valid placement is possible, then falls back to boundary placement.

A high `forced_patch_tiles` value is especially interesting because it suggests that local geometry may be forcing larger structures.

### 4. Exact Hex Region Test

The validator can also try to exactly fill finite hex-shaped regions.

Relevant output fields:

```text
exact_region_found
exact_region_radius
exact_region_cells
exact_region_limit_hit
exact_region_nodes
```

If exact regions are found, that gives additional evidence that the tile can cover structured finite areas. However, this is still finite evidence, not a proof of infinite tiling.

## Classification Labels

The validator sorts results into classification folders.

### `periodic_reject`

A periodic tiling was found. This candidate should be rejected as an einstein candidate.

### `weak_patch_reject`

No periodic tiling was found in the finite test, but the patch growth was weak.

### `unstable_candidate`

The candidate survived some tests but did not reach the stronger patch threshold.

### `strong_patch_candidate`

The candidate survived the finite periodic test and grew a large patch. These are worth saving and reviewing.

### `very_strong_candidate`

The candidate grew a very large patch and also passed additional exact-region criteria. These are the strongest finite-search candidates and should be investigated further.

## Compilation

Compile with `nvcc`:

```bash
nvcc -O3 -std=c++17 -arch=sm_80 --use_fast_math \
  einstein_validator_v1.cu \
  -o einstein_validator_v1
```

If using the Makefile:

```bash
make
```

## Input Format

The validator expects candidate files named like:

```text
candidate_0000858.txt
candidate_0000148.txt
candidate_0001633.txt
```

It does **not** validate SVG files directly.

Correct input:

```text
candidate_0000858.txt
```

Incorrect input:

```text
candidate_0000858_tile.svg
candidate_0000858_best_patch.svg
candidate_0000858_validation.txt
```

A candidate file should contain the normalised tile cells, for example:

```text
normalized_cells x y orientation:
0 4 0
0 4 1
1 1 0
...
```

To validate one candidate, place only that candidate file into its own input folder.

Example:

```bash
mkdir -p single_0858_deep
cp candidate_0000858.txt single_0858_deep/
```

## Example: Fast Batch Validation

This is useful for quickly filtering many candidates:

```bash
./einstein_validator_v1 \
  --input candidate_records \
  --out validation_fast \
  --period 16 \
  --nodes 5000000 \
  --patch-radius 96 \
  --random-trials 300 \
  --boundary-trials 250 \
  --forced-trials 250 \
  --region-min 3 \
  --region-max 5 \
  --region-nodes 1000000 \
  --weak-tiles 120 \
  --strong-tiles 300 \
  --very-strong-tiles 700
```

## Example: Medium Batch Validation

This is slower but more reliable than the fast triage run:

```bash
./einstein_validator_v1 \
  --input candidate_records \
  --out validation_medium \
  --period 18 \
  --nodes 20000000 \
  --patch-radius 128 \
  --random-trials 1000 \
  --boundary-trials 700 \
  --forced-trials 700 \
  --region-min 3 \
  --region-max 8 \
  --region-nodes 5000000 \
  --weak-tiles 180 \
  --strong-tiles 500 \
  --very-strong-tiles 900
```

## Example: Deep Validation of One Candidate

This is intended for a single strong candidate only. It can take many hours or days.

```bash
mkdir -p single_0858_deep
cp candidate_0000858.txt single_0858_deep/

./einstein_validator_v1 \
  --input single_0858_deep \
  --out validation_0858_deep \
  --period 24 \
  --nodes 200000000 \
  --patch-radius 220 \
  --random-trials 5000 \
  --boundary-trials 4000 \
  --forced-trials 4000 \
  --region-min 3 \
  --region-max 11 \
  --region-nodes 50000000 \
  --weak-tiles 300 \
  --strong-tiles 1000 \
  --very-strong-tiles 2500
```

## Running Long Jobs

Long validations should be run inside a terminal multiplexer such as `tmux`.

Create a session:

```bash
tmux new -s validate_0858
```

Run the validator command inside the session.

Detach without stopping the program:

```text
Ctrl+b
d
```

Reattach later:

```bash
tmux attach -t validate_0858
```

List sessions:

```bash
tmux ls
```

## Monitoring Jobs

Check running validator processes:

```bash
ps -u "$USER" -o pid,etime,time,%cpu,%mem,cmd | grep einstein_validator_v1
```

A live CPU-bound validator usually shows close to:

```text
99.7% CPU
```

The validator initialises a CUDA context, so it may appear in `nvidia-smi`, but the deep exact searches are mostly CPU-heavy.

Check generated files:

```bash
find validation_0858_deep -type f -ls
```

Read the CSV summary:

```bash
cat validation_0858_deep/validation.csv
```

## Interpreting a Strong Result

A promising result looks like:

```text
classification: strong_patch_candidate
periodic_found: 0
periodic_limit_hit: 0
forced_patch_tiles: high
best_patch_method: forced
best_patch_tiles: high
```

For example:

```text
periodic_found: 0
periodic_limit_hit: 0
forced_patch_tiles: 2948
best_patch_method: forced
best_patch_tiles: 2948
```

This means the candidate survived the finite periodic search and grew a large forced patch.

It does **not** prove the tile is an einstein tile.

## What Counts as Proof?

This validator provides finite computational evidence only.

To prove that a tile is an einstein tile, further work would be needed, such as:

* a mathematical proof that the tile can tile the infinite plane;
* a proof that no periodic tiling exists at any scale;
* a substitution structure;
* a forced hierarchy argument;
* an independently verifiable certificate or proof system.

## Notes

* The validator can run for many hours or days on deep settings.
* `--period` and `--nodes` have a major effect on runtime.
* Larger `--period` values greatly increase the search space.
* Deep validation should only be used for the strongest candidates.
* The safest workflow is staged testing: fast batch validation, medium validation, then deep single-candidate validation.
